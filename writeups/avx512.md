## Quick Review
- Many ternary weights {-1, 0, 1} can fit into a byte.
- We can treat groups of weights as indices into a lookup table.
- We DON'T have to unpack them for inference.

## Motivation
**How many trits can we fit into a byte?** \
5 Trits! $3^5 = 243 < 256$

**How does Bitnet pack trits?** \
3 trits per 5 bits.

**Why? That's noticeably less dense. 8/5 = 1.6 and 5/3 = 1.66666. It seems like a strange choice of numbers.** \
The reason is that AVX2 makes it harder to do bigger lookup tables.

**Okay, then how are we going to make this work?** \
AVX-512 to the rescue! Big vectors big results!

## First some spicy results!
The headline result, on one core of a Ryzen 9 9950X (Zen 5), at
$M=256$, $K=2080$, $N=2048$:

| Implementation | Gop/s | vs dense |
|----------------|------:|---------:|
| dense int8 GEMM (MSVC auto-vectorized AVX2) | 84 | 1.0x |
| BitNet TL2 (AVX2 `pshufb` kernel) | 182 | 2.2x |
| BitNet TL2 widened to 512-bit (our port) | 343 | 4.1x |
| dense int8 with AVX-512 VNNI (`vpdpbusd`) | 555 | 6.6x |
| **ours: 5 trits/byte + AVX-512 lookup** | **704** | **8.4x** |

Every implementation in the table produces bit-identical int32 outputs,
and the timed region is just the integer matmul — both BitNet kernels
accumulate int32 directly, with no float quantize/dequantize on either
side, so the comparison is like for like.

## Limitations of AVX2

AVX2's useful lookup instruction is `vpshufb`. It performs byte lookups,
but each 128-bit lane can only index a 16-entry table. That small table
budget determines Microsoft's
[BitNet](https://github.com/microsoft/BitNet) TL2 layout.

Three ternary weights have $3^3 = 27$ possible values. That does not fit
in 16 entries, but ternary dot products have a mirror symmetry:

$$
T[-c] = -T[c].
$$

So the 27 values reduce to 14 non-negative magnitudes plus a sign. That
fits `vpshufb`. The cost is that TL2 groups only three trits per lookup,
stores signs in a separate bitstream, and has to split int16 table values
into byte-sized pieces because `vpshufb` returns bytes.

This is good engineering for AVX2. It also shows the wall: with a
16-entry byte table, the hardware strongly prefers small lookup codes.

## What AVX-512 Changes

AVX-512BW gives us a different lookup primitive: `vpermt2w`, exposed in
intrinsics as `_mm512_permutex2var_epi16`. It indexes 16-bit words from a
table spanning two 512-bit registers. One such permute covers 64 entries;
two permutes plus a blend cover 128 entries.

Now redo the packing arithmetic:

| Trits per index | Combinations | Magnitudes after mirror symmetry | Fits? |
|---:|---:|---:|---|
| 3 | 27 | 14 | 16-entry AVX2 `pshufb` |
| 5 | 243 | **122** | **128-entry AVX-512 pair** |
| 6 | 729 | 365 | no |

This is the important coincidence:

$$
122 \le 128.
$$

Five ternary weights are the densest byte-aligned ternary packing:
$3^5 = 243$ values fit inside one byte's 256 codes. Mirror symmetry cuts
the lookup table to 122 magnitudes, and that fits inside the largest
two-permute AVX-512 table budget.

That means each packed byte can stay packed all the way through the
kernel. It is loaded, used as an index into register-resident int16
tables, sign-corrected, and accumulated. There is no unpacking back to
individual ternary weights.

The core lookup looks like this, simplified to one row and 32 output
columns:

```c++
__m512i  v    = _mm512_cvtepi8_epi16(packed_bytes);     // 32 packed codes
__mmask32 neg = _mm512_movepi16_mask(v);                // sign bit
__m512i  m    = _mm512_abs_epi16(v);                    // magnitude index
__mmask32 hi  = _mm512_test_epi16_mask(m, c64);         // index >= 64

__m512i r0 = _mm512_permutex2var_epi16(T[0], m, T[1]);  // entries 0..63
__m512i r1 = _mm512_permutex2var_epi16(T[2], m, T[3]);  // entries 64..121
__m512i r  = _mm512_mask_blend_epi16(hi, r0, r1);
r = _mm512_mask_sub_epi16(r, neg, zero, r);             // apply sign
acc = _mm512_add_epi16(acc, r);
```

Those 32 lookups correspond to 160 ternary multiply-adds. The lookup path
does not touch memory after the packed weights are loaded; the tables are
in zmm registers.

## Masks Matter Too

The permute is the main event, but AVX-512 mask registers remove two
pieces of overhead that would otherwise be awkward.

First, the sign of the balanced-ternary code is already present in the
packed byte. After widening to int16, `vpmovw2m` extracts those sign bits
into a mask register. A masked subtract then computes `0 - r` only for
negative lanes. The sign handling is not a separate side stream.

Second, the same vector body handles the final `N % 32` columns with
masked loads and stores. There is no scalar cleanup loop for odd column
counts. In the benchmark harness, `N=2047` stays within about 5% of
`N=2048`.

## Building The Table Cheaply

Each group of five activations needs a table of 122 int16 magnitudes. A
scalar build of those 122 entries is correct but too expensive at this
throughput; the first version of this kernel spent about 40% of its
runtime just making tables.

The table has structure. A five-trit code can be split into its top three
trits and bottom two trits:

$$
T[n] = H[\lfloor n/9 \rfloor] + L[n \bmod 9].
$$

`H` has 27 entries and `L` has 9. Both are linear in the five activation
values, so the kernel builds them with broadcasts, 16-bit multiplies, and
adds. The final 122 table lanes are assembled with constant-index
`vpermw` operations. There is no scalar per-entry loop and no division in
the hot path.

The final kernel also uses two small bookkeeping choices:

- It processes four activation rows at a time, so the decode of packed
  weight bytes is shared across rows.
- It accumulates partial sums in int16 and flushes to int32 every 51
  groups. The bound is safe because one table entry is at most
  $5 \times 128 = 640$, and $51 \times 640 = 32640 < 32767$.

These details are not the conceptual breakthrough, but they are the
difference between a neat instruction trick and a fast kernel.

## Is It The Width Or The Design?

A fair question: is this kernel faster than BitNet's simply because
512-bit registers are twice as wide? To separate width from design, we
also measured BitNet's TL2 ported up to 512-bit and our 5-trit kernel
ported down to 256-bit - all four bit-exact, both designs using matched
sweep implementations (Gop/s at $K=2080$; this controlled comparison
predates the table-build and blocking optimizations above, which is why
ours reads 355 here rather than the headline 704):

| | 256-bit | 512-bit | width scaling |
|---|---:|---:|---:|
| TL2, 3 trits, `pshufb` | 182 | 343 | 1.88x |
| ours, 5 trits, two-permute lookup | 155 | 355 | 2.29x |

At 256-bit TL2 is about 17% ahead. At that width no shuffle can index 122
entries cheaply - our lookup needs a four-permute, three-blend select
tree - and TL2's small cheap tables are the better trade. What flips the
result is how the two designs scale: doubling the register width helps
TL2 by the usual sublinear amount (1.88x), but helps the big-table design
superlinearly (2.29x), because a two-register shuffle's table capacity
doubles along with its lane count. Exactly when the lanes double, the
select tree collapses to the two permutes and one blend shown above, and
the five-trit format pulls level — a dead heat at 512-bit, within a few
percent.

So AVX-512 does not merely make the five-trit format faster; it is what
makes it competitive at all. At 256 bits TL2 is the better design, and it
remains the right choice for AVX2-only hardware. The five-trit format's
real edge shows up not here but at large matrices, against the dense VNNI
baseline below — its 5x weight compression.

## Why Dense VNNI Does Not End The Story

AVX-512 VNNI is the strongest dense int8 baseline on this CPU. The
instruction `vpdpbusd` computes packed int8 dot products directly, and it
is exactly the sort of instruction one would expect to beat a lookup
scheme.

For small enough matrices, the LUT kernel still wins by about 25%:

| $K \times N$ | dense weights | packed weights | VNNI Gop/s | LUT Gop/s | LUT advantage |
|---|---:|---:|---:|---:|---:|
| 2080 x 2048 | 4.3 MB | 0.85 MB | 555 | 704 | 1.27x |
| 4160 x 4096 | 17 MB | 3.4 MB | 485 | 718 | 1.5x |
| 8320 x 8192 | 68 MB | 13.6 MB | 127 | 690 | 5.4x |
| 16640 x 16384 | 273 MB | 54.5 MB | 123 | 582 | 4.7x |

The reason is not that VNNI is weak. It is that dense int8 weights are
five times larger than the packed ternary weights. While they still fit
this CPU's 32 MB L3 - through $K=4160$, where the dense matrix is 17 MB -
VNNI keeps pace and the gap stays near 1.5x. At $K=8320$ the 68 MB of
dense weights spill to DRAM while the 13.6 MB packed form still fits in
cache, and VNNI falls to memory speed: the LUT kernel keeps doing
register lookups over compressed weights and pulls 5.4x ahead.

The effect is starkest at $M=1$ - the single-row GEMV shape that
token-by-token inference actually runs, where everything is weight
bandwidth: 426 Gop/s for the LUT kernel against 77 for VNNI, a 5.5x
gap.

That makes the compression from Part 1 more than a storage feature. It
is a performance feature: it keeps the hot path closer to compute-bound
at sizes that resemble real model layers.

## The AVX-512 Takeaway

The important AVX-512 features are not just wider registers.

`vpermt2w` gives a 64-entry, int16, register-resident lookup primitive.
Two of them plus a blend give exactly enough table space for the 122
mirror-reduced magnitudes of five packed trits.

Mask registers make the balanced-ternary sign cheap and let the same SIMD
body handle tails.

Constant-index `vpermw` makes the per-activation table build cheap enough
to disappear into the rest of the kernel.

Together, those pieces turn the Part 1 lookup-table idea into a
single-core 704 Gop/s kernel: 3.9x faster than Microsoft's AVX2 BitNet
TL2 kernel on the same shape, and 1.27x to 5x faster than dense AVX-512
VNNI depending on matrix size, while storing weights at 1.6 bits each.
The repository also includes row-parallel multithreaded results, where
the same kernel reaches about 3.1-4.4 Top/s at 8 cores before memory
bandwidth takes over.
