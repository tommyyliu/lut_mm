*Part 1 ([Matrix Multiply using Lookup Tables](ternary-mm)) packed five
ternary weights into one byte and turned matrix multiplication into table
lookups. This post is about making those lookups fast: what breaks when you
vectorize the idea naively, how Microsoft's [BitNet
kernels](https://github.com/microsoft/BitNet) solve it within AVX2, and
which AVX-512 instructions let us go past them. Code at
[lut_mm](https://github.com/tommyyliu/lut_mm).*

The punchline first. One core of a Ryzen 9 9950X (Zen 5), $M=256$,
$K=2080$, $N=2048$, int8 activations, every kernel producing bit-identical
int32 outputs:

| Implementation | Gop/s | vs dense |
|----------------|------:|---------:|
| dense int8 GEMM (MSVC auto-vectorized AVX2) | 85 | 1.0x |
| BitNet TL2 (their AVX2 `pshufb` kernel) | 164 | 1.9x |
| BitNet TL2 widened to 512-bit (our port) | 291 | 3.4x |
| dense int8 with AVX-512 VNNI (`vpdpbusd`) | 554 | 6.6x |
| **ours: 5 trits/byte + `vpermt2w` lookups** | **695** | **8.2x** |

4.2x the official AVX2 implementation, and — more surprisingly — 1.27x
*faster than dense VNNI*, the instruction AMD built specifically for int8
dot products. We'll come back to why that gap exists and why it grows to
5x at larger sizes.

## The problem: lookups don't vectorize

The Part 1 scheme costs one table lookup and one add per packed byte. On
paper that's 5 multiply-adds for the price of 2 operations. In practice a
scalar loop does roughly one lookup per cycle, and a modern core can do
*sixty-four* int8 multiply-adds per cycle. An idea that wins on operation
count loses by an order of magnitude on operation *width*.

The obvious vector fix is the gather instruction: `vpgatherdd` fetches 8
table entries from memory with one instruction. We tried it. It exactly
ties the scalar loop — about 47 Gop/s, *half* of the naive dense GEMM —
because gather on this microarchitecture is microcoded to roughly one
lookup per cycle, same as scalar loads. The table sits in L1 and it does
not matter.

The lesson that shapes everything else: **a lookup table is only fast if
it lives in registers and is indexed by a shuffle instruction.** Shuffles
run one-per-cycle too — but each one does 32 or 64 lookups.

## What Microsoft did with AVX2

AVX2 has exactly one in-register lookup: `vpshufb`. It performs 32 byte
lookups per instruction, but each 128-bit lane indexes its own copy of a
**16-entry, byte-valued** table. Those two limits — 16 entries, 8-bit
results — dictate BitNet's whole format:

- **16 entries** caps the index at 4 bits. Three trits have $3^3 = 27$
  combinations — too many — but the mirror trick from Part 1
  ($T[-c] = -T[c]$) cuts 27 to 14 magnitudes, which fits. So TL2 packs 3
  trits as a 4-bit table index plus a sign bit stored in a separate
  bit-plane: 5 bits per 3 weights, 1.667 bits/weight.
- **Byte results** are too narrow for a dot product of int8 activations,
  so every entry is split across *two* 16-byte tables (low byte, high
  byte). Each lookup takes two `vpshufb`s plus an unpack to reassemble
  int16 values.
- The **signs** live in their own bitstream and are applied by shifting
  each bit up to position 15, arithmetic-shifting it back down into a
  0/−1 mask, and doing an xor-and-add conditional negate — about four
  instructions per group of lookups.

This is good engineering against a hard constraint, and it beats the
dense GEMM by 2x. But the constraint is real: 3 trits per lookup, two
shuffles plus reassembly per lookup, and a side-channel of sign bits.

## What AVX-512 changes: the table budget

AVX-512BW has `vpermt2w`: a shuffle that indexes a **64-entry table of
16-bit words** (the table spans two 512-bit registers) and performs 32
lookups per instruction. Two of these plus a blend cover a 128-entry
index space.

Now redo BitNet's arithmetic with a budget of 128 instead of 16:

| Trits per index | Combinations | Magnitudes after mirror trick | Fits? |
|---:|---:|---:|---|
| 3 | 27 | 14 | 16-entry `pshufb` ✓ |
| 5 | 243 | **122** | **128-entry `vpermt2w` pair ✓** |
| 6 | 729 | 365 | ✗ |

This is the coincidence the whole project rests on: $122 \le 128$. Five
trits per byte — the densest byte-aligned packing of ternary weights —
lands *just* inside the largest table AVX-512 can shuffle in two
instructions. Each lookup now resolves a 5-element dot product, not a
3-element one, and because `vpermt2w` shuffles 16-bit words natively,
the entries are full int16 values: no split tables, no unpack, no
reassembly.

The inner loop, slightly simplified (32 output columns per iteration,
`T[0..3]` holding the 122 magnitudes):

```c++
__m512i  v    = _mm512_cvtepi8_epi16(packed_bytes);     // widen 32 codes
__mmask32 neg = _mm512_movepi16_mask(v);                // sign bit -> mask
__m512i  m    = _mm512_abs_epi16(v);                    // index = |code|
__mmask32 hi  = _mm512_test_epi16_mask(m, c64);         // index bit 6

__m512i r0 = _mm512_permutex2var_epi16(T[0], m, T[1]);  // entries 0..63
__m512i r1 = _mm512_permutex2var_epi16(T[2], m, T[3]);  // entries 64..121
__m512i r  = _mm512_mask_blend_epi16(hi, r0, r1);       // T[|code|]
r = _mm512_mask_sub_epi16(r, neg, zero, r);             // apply sign
acc = _mm512_add_epi16(acc, r);                         // accumulate
```

Nine instructions for 32 lookup-accumulates, each replacing a 5-element
dot product: 160 multiply-accumulates' worth of work. The kernel runs at
the shuffle ports' throughput limit; there is no memory traffic in the
lookup path at all.

## Mask registers do the signs and the tails

The snippet above quietly uses the other half of AVX-512: the `k` mask
registers, which turn two of BitNet's instruction sequences into single
instructions.

**Signs.** Balanced-ternary codes carry their own sign — no separate
bit-plane needed. `vpmovw2m` lifts the 32 sign bits into a mask register
in one instruction, and `vpsubw` *with that mask* computes $0 - r$ only
in the negative lanes, passing the rest through. BitNet's
shift/shift/xor/add dance becomes extract-mask, masked-subtract.

**Tails.** When $N$ isn't a multiple of 32, the last columns run the
*same* SIMD body under `_mm512_maskz_loadu_epi8` / masked stores. There
is no scalar cleanup loop anywhere in the kernel, which is why $N=2047$
runs within ~5% of $N=2048$ instead of falling off a cliff.

## Building 122 entries without 122 dot products

Part 1 waved this off as "built once per group and reused across all
columns." True, but at these speeds even the build shows up: a scalar
build (122 five-element dot products per group of activations) was 40%
of total runtime in our first version.

The fix is that the table is *structured*: entry $n$ (for
$n = |v| + 121$) decomposes by place value into

$$
T[n] = \underbrace{H[\lfloor n/9 \rfloor]}_{\text{top 3 trits}} +
       \underbrace{L[n \bmod 9]}_{\text{bottom 2 trits}}
$$

$H$ has 27 entries, $L$ has 9, and both are linear in the activations:
$H = a_0 D_0 + a_1 D_1 + a_2 D_2$ where the $D_i$ are *constant* digit
vectors. So the build is five broadcasts, five 16-bit multiplies, three
adds — and then the four table registers are assembled by `vpermw` with
*compile-time-constant* indices ($\lfloor n/9 \rfloor$ and $n \bmod 9$
for each lane are known statically). No scalar work, no division, ~18
instructions per group. The build dropped from 40% of runtime to noise;
this change plus the row blocking and flush scheduling below took the
kernel from 355 to 695 Gop/s.

## Two bookkeeping tricks

**Accumulate in int16, flush late.** A table entry is at most
$5 \times 128 = 640$ in magnitude, so $51$ groups can accumulate in
int16 without overflow ($51 \times 640 = 32640 < 32767$). Accumulating
at half width doubles the lanes per add; every 51 groups (255 weights)
the partials widen and flush into the int32 output, and the *first*
flush stores instead of adds so the output never needs pre-zeroing.

**Share the decode across rows.** Everything computed from the packed
bytes — widening, absolute value, both masks — depends only on the
weights, not the activations. Processing 4 activation rows per sweep
amortizes that decode 4 ways; each extra row pays only its two permutes,
blend, masked subtract, and add. (Eight rows would need 32 table
registers, which spill and give the win back.) Net cost: just over 8
µops per 32 outputs per row.

## Width scaling is superlinear for big tables

Is the win just "512 bits is twice 256"? No — and this is the most
interesting measurement in the project. We ported BitNet's TL2 design
to 512-bit and our design down to 256-bit, all four bit-exact (Gop/s at
$K=2080$):

| | 256-bit | 512-bit | scaling |
|---|---:|---:|---:|
| TL2, 3 trits, `pshufb` | 163 | 292 | 1.79x |
| ours, 5 trits, `vpermt2w` | 155 | 355 | **2.29x** |

At 256-bit the designs *tie* — at that width no shuffle can index 122
entries cheaply, so our kernel pays a 4-permute select tree and TL2's
cheaper lookups are the better trade. Doubling the registers helps TL2
sublinearly (1.79x, the usual story). It helps the big-table design
*superlinearly*, because a two-register shuffle's table capacity doubles
*along with* the lane count: the select tree collapses to 2 permutes +
1 blend exactly when the lanes double. AVX-512 doesn't just make the
5-trit format faster — it's what makes it viable at all. TL2 remains the
right design for AVX2-only hardware.

## The dense ceiling: beating VNNI

The fairest possible dense baseline is AVX-512 VNNI: `vpdpbusd` computes
16 four-element int8 dot products per instruction, on unpacked weights.
Surely dedicated dot-product silicon beats table lookups?

| $K \times N$ | dense weights | packed | VNNI Gop/s | LUT Gop/s | LUT advantage |
|---|---:|---:|---:|---:|---:|
| 2080 × 2048 | 4.3 MB | 0.85 MB | 554 | 702 | 1.27x |
| 4160 × 4096 | 17 MB | 3.4 MB | 347 | 714 | 2.1x |
| 8320 × 8192 | 68 MB | 13.6 MB | 123 | 639 | **5.2x** |
| 16640 × 16384 | 273 MB | 54.5 MB | 110 | 482 | 4.4x |

Even fully cache-resident, the LUT kernel wins by 27% — one lookup
retires 5 weights against `vpdpbusd`'s 4, with no unsigned-operand bias
correction. But the real story is the second column: dense weights are
5x the bytes. At $8320 \times 8192$ they fall out of L3 while the packed
weights still fit, and VNNI craters to memory speed while the LUT kernel
barely moves. Once both spill (last row), the ratio settles near the 5x
density advantage itself. At GEMV ($M=1$, the shape token-by-token
inference actually runs) the same effect is starker still: 410 vs 76
Gop/s, a 5.4x win, because single-row inference is pure weight
bandwidth.

The compression from Part 1 turns out not to be a storage feature. It is
the *performance* feature: it is what keeps the kernel compute-bound at
the sizes where LLM layers actually live.

## Summary

What AVX-512 contributes, beyond doubling the registers:

1. **`vpermt2w`** — a 64-entry, word-granularity, two-register shuffle.
   Its 128-entry pair budget admits 122 mirror-reduced magnitudes of 5
   packed trits, vs `pshufb`'s 16-entry budget admitting 14 magnitudes
   of 3. More trits per lookup, int16 entries natively, no table
   reassembly.
2. **Mask registers** — `vpmovw2m` + masked subtract apply balanced
   ternary's built-in signs in 2 instructions; masked loads/stores
   eliminate scalar tails.
3. **Constant-index `vpermw`** — assembles the structured table from two
   small linear pieces, ~18 instructions per group, no scalar build.

Together: 695 Gop/s on one core, bit-exact against the dense reference,
4.2x Microsoft's AVX2 kernels and 1.3–5x dense VNNI depending on size —
with the weights at 1.6 bits each the entire time. Multithreaded scaling
(rows parallelize trivially; ~3.1–4.4 Top/s at 8 cores before memory
bandwidth takes over) and full benchmark methodology are in the
[repository](https://github.com/tommyyliu/lut_mm).
